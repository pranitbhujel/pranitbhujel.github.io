Getter and Setter methods¶

The purpose of getter and setter methods is to restrict the access to the variables directly. Languages like Java make it compuslory for us to write the methods, while Python gives us the flexibility of writing these methods whenever we want to do so.

Explicitly writing getter and setter can be done as well but the approach involving decorators is recommended in Python. While not using the decorators, the problem of backward compatibility occurs. Let's see how it happens.

In [53]:
class Rectangle:
    def __init__(self, width, height):
        self.width = width
        self.height = height
        
    def get_width(self):
        return self.width
    
    def set_width(self, width):
        self.width = width
    
    def get_height(self):
        return self.height
    
    def set_height(self, height):
        return self.height

In [55]:
r.height
Out[55]:
20

This one works well, but people will also be able to use the obj.height to access the variable. Since python doesn't mandate the getter and setter, so this solution will not work.

. . .¶

We also know that the variable that has single underscore in the front is supposed to be a private variable, but it is only to warn the user that the variable is private. It doesn't actually restrict the user from not using the variable to modify it directly. Also, if we change the variable to say _width, then people having written their program as self.width will face issues.

So what do we have now? Decorators to the rescue!

In [1]:
class Rectangle:
    def __init__(self, width, height):
        self._width = width
        self._height = height
        
    @property
    def width(self):
        return self._width
    
    @width.setter
    def width(self, width):
        if width <= 0:
            raise ValueError('Width must be greater than 0.')
        else:
            self._width = width
    
    @property
    def height(self):
        return self._height
    
    @height.setter
    def height(self, height):
        if height <= 0:
            raise ValueError('Height must be greater than 0.')
        else:
            self._height = height
In [2]:
r = Rectangle(10, 20)

r.width
Out[2]:
10

So while using decorators, we can see that we did not have to use the parenthesis after the end of the width, which now is actually a function!

Also since the name is simply width and no underscores upfront or no parenthesis at the back, backwards compatibility is ensured as well. We just have to get or set it using .width.

Multi-Line Statements and Strings¶

Understanding Multi-Line statements definitely makes us a better developer. Multi-Line statements/strings are generally used for improving the readability of the code. The interpreter doesn't bother if the given program is cramped in one line or extended to multiple lines, as long as the syntax is right.

First, we need to understand the difference between physical newline character and logical newline token.

. . .¶

Physical Newline Character: '\n' The character which is passed onto the program while we press enter to write a statement in the newline.

Logical Newline Token: Once we execute the program, the python interpreter breaks the program into logical lines of code and they are tokenized. These tokens/logical lines end with a logical newline token, which is then used by the python interpreter to execute the code.

Sometimes, physical newlines are ignored in order to combine multiple physical lines terminated by a logical newline token.

Now that we know what physical and logical newlines are, we can move forward to ways, the compiler converts physical newlines to logical newlines.

. . .¶

Conversion can be implicit or explicit. Implicit is when the compiler automatically converts to logical and Explicit is when we explicitly put '\' character to tell the compiler

The General Notion:¶

Variables/Identifiers are case sensitive and can have letters, _, and digits. It can begin with letters and _, but not with a digit.

Single Underscore in the Front¶

This is a convention to indicate that the identifier is to be used for internal use or private objects. This kind of objects will not get imported by an import statement such as from module import *.

However there are other ways of accessing it.

. . .¶

Double Underscore in the Front¶

This convention is primarily used for something called as Name Mangling. Name Mangling is a process in which an identifier with a double underscore in the front __name is replaced with _ClassName__name, and we can access the variable using that name.

It allows us to access the class variables outside the class (in sub-classes), mostly used in inheritance.

To know more about the Name Mangling, Click here

. . .¶

Double Underscore in the Front and the Back¶

This convention is used for system-defined names that have a special meaning to the interpreter. For an example __init__ or __main__. Defining your own name may confuse the interpreter to cause unncecessary errors and should avoid doing that.

Sticking to the ones pre-defined by Python is certainly a good idea!

. . .¶

Other Naming Conventions¶

Packages: short, all-lowercase names, preferably no underscores. Eg: package

Modules: short, all-lowercase names, can have underscores. Eg: a_module amodule

Classes: CapWords (upper camel case) convention. Eg: BankAccount

Functions: lowercase, words separated by underscores (snake_case) Eg: open_account

Variables: lowercase, words separated by underscores (snake_case) Eg: my_name, _name, _

Constants: all-uppercase, words separated by underscores Eg: NEW_CONSTANT

Garbage Collection¶

Circular Reference: When there's a circular reference that keeps the reference count greater than 0 and so the reference count mechanism is not always suitable to free the unused objects. In this case the garbage collector does it work.

It is not recommended to use Python V3.4 and before, as the python interpreter gets confused over this criteria; If __del()__ destructor is defined in the class of the object that is stuck in the circular reference, the gc would get confused as in what sequence should it follow to destroy those objects and hence would not be able to destroy!

This causes the object to be marked as uncollectable and the objects in the circular reference are not cleaned up - a term called Memory Leak.

It was however solved in the later python versions. Now, garbage collector runs periodically to sweep extra objects which are not necessary for the program and hence consumes a good chunk of computational power.

Also, the garbage collector can be disabled using the gc module in python. However, it is not recommended as the memory allocation is pretty unpredictable, so it should not be disabled unless you're absolutely certain of what you're doing! By default the garbage collector is turned on.

Statically Typed vs Dynamically Typed¶

If we look at some programming languages like C, C++ and Java, we'll notice that the variable declaration has three parts:

Statically Typed:

int a = 10;

Now, if we look into Python we just have to write

Dynamically Typed:

x = 10

So the only difference is that, in statically typed languages, we explicitly define the variable to be of a certain type. Hence if we try to allocate a value other than that particular data type, it will throw an error.

However in dynamically typed language, the variable is merely a reference to the memory location and hence we can change it to point to any other memory location and it won't throw an error!

Shared References and Mutability¶

Python Memory Manager is smart, in the sense that when it creates a shared reference to a memory address, it shares the same location with the immutable objects but creates different memory location for the mutable objects.

The good reason for doing that is, say we define a=[1, 2, 3] and we the write b=a, and now if we change b by adding or removing any element then it'll affect a as well.

So when mutable objects are created, the python memory manager creates them at different memory location even if they have the same content.

Interning - The Power of Optimization!¶

This is a very cool concept. When we tried creating two variables and checking their memory address using id() function, despite being immutable objects we noticed that for a=500 and b=500, we had different memory addresses. Why?

Interning - Interning is the process through which python interpreter/memory manager caches some integer or string objects at the start-up. The integer value starts at - 5 to 256, so that when we create an integer, we reduce the memory overhead and the cached object is returned.

That is why when we create something bigger than 256, we get a different memory location even though immutable. Now, what's the use of string interning and how can we do it manually?

The string when we intern it using sys.intern(string), we're basically caching it. So the fetching and execution becomes a lot faster when we're dealing with a lot of string values. A process indeed helpful in Natural Language Processing!

Also interning makes it easy for the string comparison as well because when comparing string we've to compare them character by character. Now, when we intern the strings then we know that if they're same then they will be stored in the same memory location. And the power of interning allows us to compare the string using just their memory address - using the identity operator is or is not!

As long as the string looks like an identifier, it'll get interned automatically. Also pointing to an already referenced object is much faster than creating a new object in the memory!

PeepHole Optimization¶

This is done in two different ways.

  1. Constant Expressions
  2. Transforming mutable objects to immutable.

Constant Expressions are evaluated at the compilation time. So when there are constant expression that do not change over the whole program is evaluated and then saved for faster reference.

Also mutable objects like lists are converted to immtable tuple. Similarly sets are converted into frozen sets, which makes the membership operations much faster compared to lists or tuples.

Remember that for upto 20 chars or 20 elements of any objects, the expression is saved as constant expression. This way the execution time becomes much shorter.

Floor and Modulo Division - The equation!¶

Now, probably the most important concept. Why does the floor and modulo operation return different numbers on the sign change, while operating on the same number?

Remember we can break a division operation into an equation like this.

nominator / denominator = denominator * quotient(nom // denom) + remainder(nom % denom)

So, whenever we do those two operation, it always obeys the equation above. That is why we get different values while doing those operations for the postive and negative operands.
